Наблюдаемость LLM-приложений (LLM Observability)

Наблюдаемость (observability) — это способность понимать внутреннее состояние системы
по её внешним выходам. Для LLM-приложений это означает возможность отслеживать,
анализировать и отлаживать каждый вызов языковой модели и связанные операции.

Три столпа наблюдаемости LLM:

1. Трассировка (Tracing)
Трассировка позволяет отслеживать полный путь запроса через систему.
В контексте LLM это включает: входной промпт, параметры модели, сгенерированный ответ,
время выполнения, количество использованных токенов и стоимость.
Трассировка особенно важна для сложных пайплайнов, где один запрос пользователя
проходит через несколько этапов: предобработка, извлечение контекста (RAG),
генерация ответа, постобработка.

2. Метрики (Metrics)
Ключевые метрики LLM-приложений:
- Латентность (время ответа модели)
- Стоимость за запрос (на основе количества токенов)
- Частота ошибок (таймауты, rate limits, некорректные ответы)
- Качество ответов (через автоматические и ручные оценки)
- Throughput (количество запросов в единицу времени)

3. Логирование (Logging)
Структурированное логирование промптов, ответов и метаданных позволяет:
- Отлаживать проблемы с качеством ответов
- Воспроизводить конкретные взаимодействия
- Строить датасеты для оценки и файн-тюнинга
- Обеспечивать аудит и compliance

Зачем нужна наблюдаемость LLM?

Без наблюдаемости LLM-приложение — это "чёрный ящик". Вы не знаете:
- Почему модель дала некорректный ответ
- Сколько стоит каждый запрос
- Где узкое место в цепочке обработки
- Происходят ли утечки данных через промпты
- Пытаются ли пользователи манипулировать моделью

Инструменты наблюдаемости: Langfuse, LangSmith, Phoenix (Arize), Helicone, Braintrust.
Langfuse выделяется как open-source решение с возможностью self-hosted развёртывания,
что критически важно для организаций с требованиями к безопасности данных.

Архитектура трассировки Langfuse:

Trace (трассировка) — корневой элемент, представляющий один полный запрос.
├── Span (спан) — единица работы (предобработка, извлечение данных)
│   ├── Generation (генерация) — вызов LLM с метаданными модели и токенов
│   └── Event (событие) — точечное действие (лог, проверка)
├── Generation — может быть и на верхнем уровне
└── Score (оценка) — привязанная к трассировке или наблюдению оценка качества

Каждый элемент автоматически получает временные метки начала и конца,
что позволяет строить waterfall-диаграммы и находить узкие места.
